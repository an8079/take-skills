# 工作流规则

遵循完整的工作流程，确保每个阶段的质量。

## 工作流程

```
需求访谈 → 规格设计 → 实现计划 → 编码实现 → 测试验证 → 代码审查 → 打包交付 → 迭代优化
```

## 各阶段规则

### 需求访谈阶段

- 不要立即给出解决方案，先完全理解问题
- 主动发现盲点，指出需求中的问题
- 深度优先于速度，宁可多问几轮
- 访谈不按固定轮次结束，根据 8 维度完整度判断
- 检测是否需要创建新技能

### 规格设计阶段

- 规格必须面面俱到，编码时不应该有「这个没想到」
- 技术规格必须有足够深度，不只说「做什么」
- 必须包含风险识别和应对措施
- 必须包含验收标准

### 编码实现阶段

- 每次只做一个任务
- 完成后展示结果，等待确认
- 遇到规格外问题，回到规格阶段澄清
- 每次改动后更新项目进展

### 测试验证阶段

- 单元测试覆盖率必须 >= 80%
- 核心功能必须有集成测试
- 关键用户流程必须有 E2E 测试
- 测试失败必须修复后继续

### 代码审查阶段

- Critical 级别问题必须修复
- Important 级别问题应该修复
- 不达质量标准绝不进入下一阶段

### 交付阶段

- 必须通过安全审查
- 交付包必须包含完整文档
- .env 必须在 .gitignore 中

## 快速模式

简单任务可以使用快速模式：

- 跳过完整访谈
- 直接输出简化规格
- 用户确认后开始编码

快速模式适用场景：
- 简单的单一功能
- 用户已有明确需求描述
- 小型修复或改进

## 绝不违反的原则

1. 绝不猜测 — 不确定就问，不要假设
2. 绝不跳步 — 按工作流程顺序执行
3. 绝不过度 — 每轮只问 2-3 个问题，每次只做一个任务
4. 绝不遗忘 — 始终参照规格文档
5. 主动确认 — 每个阶段结束都要用户明确确认
