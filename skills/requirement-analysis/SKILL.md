# 需求分析技能

## When to Use This Skill

- 新项目启动时
- 需求变更时
- 功能规划时
- 用户说 `/interview` 时

## 8 维度完整度检查

访谈不按固定轮次结束，而是根据完整度判断：

| 维度 | 检查点 | 状态 |
|------|--------|------|
| 业务理解 | ✅ 清楚要解决什么问题、为谁解决、为什么现有方案不行 | ⬜ |
| 功能边界 | ✅ 明确 MVP 范围、知道哪些不做、边界情况已讨论 | ⬜ |
| 技术可行性 | ✅ 技术栈已确定、架构方向已明确、关键技术难点已识别 | ⬜ |
| 数据流 | ✅ 数据从哪来、存到哪、怎么流转、格式是什么 | ⬜ |
| 接口契约 | ✅ 对外接口已定义、输入输出已明确 | ⬜ |
| 异常处理 | ✅ 错误场景已枚举、处理策略已确定 | ⬜ |
| 非功能需求 | ✅ 性能、安全、可维护性要求已明确 | ⬜ |
| 部署交付 | ✅ 部署方式、环境要求、交付物已确定 | ⬜ |

## Quick Reference

### 访谈原则

1. **不要立即给出解决方案** — 先完全理解问题
2. **主动发现盲点** — 用户说的可能有遗漏或不合理之处
3. **深度优先于速度** — 宁可多问几轮
4. **每轮 2-3 个问题** — 避免信息过载
5. **持续记录和确认** — 每轮结束时总结

### 访谈框架

```
阶段一：问题域探索
  → 核心问题、目标用户、痛点、成功标准

阶段二：功能深挖
  → 使用场景、MVP 边界、参考产品

阶段三：技术分析
  → 技术栈、数据来源、外部集成、现有代码

阶段四：架构与数据
  → 核心模块、数据模型、状态管理

阶段五：边界与异常
  → 边界情况、误用防范、错误处理、安全

阶段六：非功能需求
  → 性能、可用性、可维护性、可扩展性

阶段七：部署与交付
  → 部署环境、交付物、CI/CD、用户水平
```

### 主动分析与质疑

在访谈过程中，必须主动：

1. **发现需求矛盾**
2. **指出潜在风险**
3. **质疑不合理假设**
4. **提出替代方案**
5. **预判技术难点**

## Examples

### Example 1: 电商项目访谈

**阶段一：问题域探索**

```
问题：
1. 这个项目要解决什么核心问题？
   答：现有电商平台缺少个性化推荐，导致转化率低

2. 目标用户是谁？他们目前怎么解决这个问题？
   答：B2C 电商平台的用户，目前没有解决方案

3. 为什么现有方案不满足需求？痛点在哪？
   答：没有基于用户行为的个性化推荐
```

**发现的问题：**
- 「个性化推荐」技术复杂度高，需要 ML 能力
- 需要收集用户行为数据，涉及隐私问题

### Example 2: 主动质疑

```
用户说：我要做一个简单的购物车

AI 主动发现的问题：
- 「简单」的标准是什么？只添加商品，还是包括数量修改、删除？
- 是否需要保存购物车？用户刷新后是否保留？
- 价格变动时如何处理？购物车中的价格是否需要更新？
```
