# AI视角模板参考

## 何时切换视角

| 场景 | 推荐视角 |
|------|---------|
| 架构决策、技术选型 | CEO视角 / 成本分析 |
| 需求模糊、方向不明 | 苏格拉底提问 |
| 技术方案评审、质疑假设 | Socratic质疑 |
| 复杂问题调试、根源分析 | investigate调查模式 |
| 项目/周/Sprint回顾 | retro KPT法 |
| 多角色并发讨论 | omc-team |
| 风险评估、应急预案 | 风险矩阵 |

## CEO视角

**触发**：架构决策、技术选型、重大方案评估。

```
# CEO视角分析

## 决策：[描述当前决策]

## 投入分析
- 💰 开发成本：[人天/月]
- 🏗️ 基础设施：[云服务/硬件]
- 📈 维护成本：[每月]

## 收益分析
- 🚀 业务价值：[具体业务收益]
- 👥 用户增长：[量化指标]
- ⏱️ 时间节省：[量化节省]

## 风险敞口
- 🔴 高风险：[具体风险] → [缓解措施]
- 🟡 中风险：[具体风险] → [缓解措施]
- 🟢 低风险：[具体风险]

## 竞争壁垒
- 护城河是什么？
- 差异化优势？

## 时间窗口
- 上线deadline：
- 关键里程碑：

## 结论
[采用 / 否决 / 暂缓] + 3句话理由
```

## 苏格拉底提问（问题根源分析）

**触发**：需求模糊、反复讨论无果、假设不明确。

```
# 苏格拉底提问分析

## 基础问题
当前观点/结论是什么？**依据是什么？**

## 假设问题
我们假设的前提是什么？
- 前提1：[是否总是成立？]
- 前提2：[是否总是成立？]
- 前提3：[是否总是成立？]

## 证据问题
支持这个结论的证据是什么？
- 证据1：[是否可靠？]
- 证据2：[是否充分？]

## 替代问题
有没有其他可能的解释/方案？
- 替代解释1：
- 替代解释2：

## 后果问题
如果这个结论/方案错了，后果是什么？
- 直接后果：
- 连锁后果：

## 根源结论
经过苏格拉底质疑后，我们真正要解决的核心问题是：
[一句话描述]
```

## Socratic质疑

**触发**：技术方案评审、代码审查、架构设计评审。

```
# Socratic质疑评审

## 方案：[技术方案名称]

## 假设检验
| 假设 | 证据支持 | 反驳 |
|------|---------|------|
| [假设1] | [证据] | [反驳] |
| [假设2] | [证据] | [反驳] |

## 反例构造
- 能找到一个反例使方案失效吗？
  → [反例描述]

## 边界条件
- 在什么条件下方案会失败？
  → [边界1]
  → [边界2]

## 最强版本
将方案修正为最强形式：
[修正后的方案]

## 结论
- [接受] — 理由
- [拒绝] — 理由  
- [修改后接受] — 需要的修正
```

## investigate 调查模式（调试/根因分析）

**触发**：Bug调试、系统故障、性能问题。

```
# 调查模式：5个为什么 + 证据链

## 问题陈述
[问题描述]

## 5个为什么（溯源）
为什么1：发生这个问题？
→ [原因1]
    为什么2：这个原因导致什么？
    → [原因2]
        为什么3：这个原因来自哪里？
        → [原因3]
            为什么4：这个根本原因是什么？
            → [根本原因]
                为什么5：这个问题暴露了什么系统缺陷？
                → [系统性缺陷]

## 证据链
| 层级 | 证据 | 来源 |
|------|------|------|
| 现象 | [观察到的现象] | [日志/用户报告] |
| 直接原因 | [直接原因] | [证据] |
| 根本原因 | [根本原因] | [证据] |

## 修复建议
1. [立即修复]
2. [短期修复]
3. [长期改进]

## 预防措施
如何防止再次发生？
```

## retro 回顾模式（KPT法）

**触发**：Sprint回顾、项目回顾、周回顾。

```
# 回顾会议：KPT法

## 会议信息
项目/阶段：
参与人：
日期：

## ✅ Keep（继续做的）
- [做得好的，要保持]
- [成功的做法]

## ⚠️ Problem（问题）
- [遇到的问题]
- [不顺畅的地方]

## 💡 Try（尝试改进）
- [下次要尝试的新方法]
- [改进计划]

## 行动项
| 行动项 | 负责人 | 截止日期 |
|--------|--------|---------|
| [行动] | [人] | [日期] |

## 结 论
本次回顾最重要的1个收获：
[一句话]
```

## omc-team 多角色讨论

**触发**：复杂决策、多个技术方案评审。

```
# omc-team 团队讨论

## 议题：[需要讨论的问题]

## 角色任命

### 🥇 激进派
立场：追求创新和速度，愿意承担风险
发言：
- 最激进的方案是什么？
- 最快的实现路径？
- 最大胆的假设？

### 🥉 保守派
立场：关注风险和稳定性
发言：
- 这个方案的最大风险？
- 什么情况下会失败？
- 现有系统能承受吗？

### ⚖️ 务实派
立场：平衡激进和保守
发言：
- 现实可行的折中方案？
- 最小可行产品是什么？
- 如何分阶段实施？

## 讨论结果
### 共识
- [已达成共识的点]

### 分歧
- [仍有分歧的点] → [提交用户决策]

### 决策建议
推荐方案：[方案名称]
理由：[3句话]
```

## 风险矩阵评估

**触发**：技术方案评估、项目规划。

```
# 风险矩阵

| 风险 | 概率 | 影响 | 风险值 | 应对策略 |
|------|------|------|--------|---------|
| [风险1] | 高/中/低 | 高/中/低 | [P×I] | [策略] |
| [风险2] | 高/中/低 | 高/中/低 | [P×I] | [策略] |

## 高风险项（红色）
- [风险] → [缓解措施] → [责任人]

## 中风险项（黄色）
- [风险] → [缓解措施]

## 低风险项（绿色）
- [风险] → [监控]
```
