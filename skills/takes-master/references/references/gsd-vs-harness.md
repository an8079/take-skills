# GSD vs Harness 工程对比

## 本质区别

| 维度 | GSD | Harness |
|------|-----|---------|
| **核心理念** | 结构化执行方法论 | 模型能力扩展 |
| **执行模式** | 强制计划-执行-验证循环 | 模型自主决定工具调用 |
| **决策者** | 人/AI协作，人主导计划 | LLM主导，Harness提供工具 |
| **透明度** | 高（每步有记录） | 中（模型决策黑盒）|
| **适用场景** | 确定性强的工程任务 | 开放性探索任务 |
| **验证方式** | 强制性四层验证 | 模型自验证或无验证 |

---

## 选择指南

```
任务类型判断
│
├─ 工程开发任务（写代码/改bug/重构）
│   └─ → GSD（结构化执行）
│
├─ 研究探索任务（调研/分析/方案评估）
│   └─ → Harness（灵活探索）
│
├─ 混合任务（先研究后开发）
│   └─ → GSD为主 + Harness辅助探索
│
└─ 复杂系统调试
    └─ → investigate调查模式 + GSD执行
```

---

## GSD的使用场景

**最适合GSD的场景：**

- ✅ 新项目初始化和规划
- ✅ 阶段式开发（明确的阶段目标）
- ✅ 需要审计轨迹的开发任务
- ✅ 多任务协作（有依赖关系的任务）
- ✅ 长周期项目（需要状态持久化）
- ✅ 政府/合规项目（需要可追溯）

**不适合GSD的场景：**

- ❌ 纯探索性任务（不知道答案在哪里）
- ❌ 一次性简单任务（5分钟能完成的）
- ❌ 创意类任务（需要发散思维）

---

## Harness的使用场景

**最适合Harness的场景：**

- ✅ 研究和调研（网页搜索+内容分析）
- ✅ 开放性问题（没有固定答案）
- ✅ 需要模型自主决策的工具调用
- ✅ 快速探索未知领域
- ✅ 单次调用的简单任务

**不适合Harness的场景：**

- ❌ 需要可审计的开发任务
- ❌ 多步骤有依赖的项目
- ❌ 政府合规项目

---

## 协同模式

### 最佳实践：GSD为主，Harness为辅

```
┌─────────────────────────────────────────┐
│  GSD主导：项目结构 + 阶段管理 + 验证     │
│                                         │
│  在GSD的各阶段内：                      │
│    - 计划阶段 → Harness探索研究         │
│    - 执行阶段 → GSD结构化执行           │
│    - 验证阶段 → GSD四层验证             │
│                                         │
│  特定任务使用Harness能力：               │
│    - investigate调查模式（调试）         │
│    - 苏格拉底提问（需求不清）            │
│    - omc-team（复杂决策）               │
└─────────────────────────────────────────┘
```

### 具体示例

| 任务 | 方法 | 原因 |
|------|------|------|
| 启动新项目 | GSD | 需要规划+结构+验证 |
| 研究技术选型 | Harness | 探索+对比+灵活 |
| 实现API | GSD | 结构化+可审计 |
| 调试复杂Bug | investigate + GSD | 根因分析+结构化修复 |
| 写测试用例 | GSD | 规范化+覆盖率目标 |
| 架构评审 | omc-team + Socratic | 多角度质疑 |
| 需求不清 | 苏格拉底提问 + GSD | 根源挖掘+规划 |

## 在本项目中的应用

针对 orchestration-service 省级AI平台开发：

**使用GSD的场景：**
- 新功能开发 → /takes:plan-phase → /takes:execute-phase → /takes:verify-work
- Bug修复 → /takes:plan-fix → /takes:execute-phase
- 项目整体管理 → /takes:progress

**使用Harness辅助的场景：**
- 研究新框架/技术 → 直接使用Harness探索
- 代码库理解 → /takes:map-codebase
- 复杂决策 → 苏格拉底提问 / omc-team

**协同模式：**
```
阶段1: 技术研究
  → Harness: 快速探索3个技术方案
  → GSD: 制定实施计划

阶段2: 开发
  → GSD: 结构化执行
  → Harness: 遇到新问题随时探索

阶段3: 验证
  → GSD: 四层验证
  → Harness: 测试覆盖率不足时探索
```
