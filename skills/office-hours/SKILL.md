# Office Hours - 结构化问题定义与产品思维

## 理念

好的开始是成功的一半。在写代码之前，正确理解问题能节省 10 倍的时间。

本技能模拟产品经理的"办公时间"——深度对话，理解真正的需求，挑战不合理的假设，探索更好的解决方案。

## 使用场景

- 用户提出一个功能需求时
- 开始一个新项目或重大重构前
- 遇到模糊的问题描述时
- 需要在多个方案中做选择时
- 准备写产品规格文档前

## 三阶段流程

```
阶段 1: 问题探索 ─── 理解真正的需求
    ↓
阶段 2: 挑战假设 ─── 质疑前提条件
    ↓
阶段 3: 方案探索 ─── 找到最佳路径
```

## 阶段 1: 问题探索

### 核心问题清单

在开始任何工作前，必须回答这些问题：

#### 1. 问题陈述
```
- 用户/客户面临什么问题？
- 这个问题多久出现一次？
- 问题的严重程度如何？
- 这个问题现在是如何解决的（如果有）？
```

#### 2. 利益相关者
```
- 谁是这个问题的主要受影响者？
- 谁会使用这个解决方案？
- 谁会反对这个解决方案？
- 谁来做最终决策？
```

#### 3. 成功标准
```
- 如何衡量解决方案的成功？
- 什么是"足够好"的标准？
- 有没有量化的指标？
- 什么时候可以认为任务完成？
```

#### 4. 约束条件
```
- 时间限制是什么？
- 预算/资源限制？
- 技术约束（必须兼容的系统）？
- 非技术约束（政策、合规）？
```

### 探索对话模板

```
🤔 问题澄清对话

问: "你能描述一下具体遇到什么问题吗？"
→ 关注具体的场景，而非抽象的功能

问: "这个问题什么时候出现？频率如何？"
→ 理解问题的普遍性和严重性

问: "如果今天这个问题神奇地解决了，你最开心的是哪一点？"
→ 理解核心价值诉求

问: "你见过别人是怎么解决类似问题的吗？"
→ 探索参考案例

问: "如果只能解决 50%，你会优先解决哪部分？"
→ 理解优先级和核心价值

问: "什么样的结果会让你失望？"
→ 理解期望和边界
```

### 痛点分类

```
🔴 高频 + 高影响 = 必须立刻解决
🟡 高频 + 低影响 = 值得优化但可等待
🟢 低频 + 高影响 = 需要优雅处理
⚪ 低频 + 低影响 = 可以暂时忽略
```

## 阶段 2: 挑战假设

### 常见错误假设

在深入之前，质疑这些常见假设：

```
❌ 假设1: 用户知道自己需要什么
→ 实际上用户描述的是解决方案，而非问题

❌ 假设2: 增加功能就能解决问题
→ 有时候减少功能、简化流程才是答案

❌ 假设3: 技术方案是瓶颈
→ 有时候流程、培训或组织变革才是真正的瓶颈

❌ 假设4: 这是新问题
→ 很多"新问题"只是老问题的变种

❌ 假设5: 我们是第一个遇到这个问题的
→ 搜索行业案例，可能已有成熟解决方案
```

### 苏格拉底式提问

```
质疑现状:
"为什么这个问题需要解决？"
"如果这个问题不被解决，会发生什么？"
"现状有什么优点？为什么不想改变？"

质疑方案:
"为什么这个方案是可行的？"
"有没有考虑过其他方案？"
"这个方案的最大风险是什么？"

质疑需求:
"这个需求是从哪里来的？"
"这个需求背后的真正目的是什么？"
"有没有更简单的方式达到同样目的？"
```

### 反向思维

```
"如果我们完全不做这件事呢？"
→ 评估不作为的后果

"如果我们只做 20% 呢？"
→ 探索最小可行方案

"如果我们必须明天上线呢？"
→ 聚焦核心功能

"如果我们有 10 倍的用户呢？"
→ 早期发现扩展性问题
```

## 阶段 3: 方案探索

### 方案生成方法

#### 1. 竞品分析
```
搜索: "[问题关键词] 如何解决"
- 行业领导者怎么做？
- 新兴公司有什么创新？
- 不同行业的类比方案？
```

#### 2. 约束放宽法
```
原始约束: 必须支持移动端
放宽: 如果只支持 Web？
更放宽: 如果是纯后端 API？

→ 也许移动端不是核心，可以 MVP 后再考虑
```

#### 3. 用户旅程法
```
从用户视角走一遍完整流程:
用户 → 发现问题 → 搜索方案 → 评估 → 使用 → 反馈

在每个步骤寻找改进机会
```

### 方案评估矩阵

```
方案评估表

标准            | 方案A | 方案B | 方案C
----------------|-------|-------|-------
实施难度        | 高/中/低 |       |
开发时间        |       |       |
用户价值        |       |       |
可扩展性        |       |       |
维护成本        |       |       |
风险            |       |       |
────────────────────────────────────
总分            |       |       |
推荐            |       |       |
```

### 决策框架

使用 **影响力/实施成本** 矩阵：

```
                    实施成本
               低          高
          ┌─────────┬─────────┐
     高   │  快速   │  战略   │
  影响    │  行动   │  项目   │
  力      ├─────────┼─────────┤
     低   │  委托   │  忽略   │
          │  他人   │         │
          └─────────┴─────────┘
```

## 与 Plan CEO Review 的关系

Office Hours 产出**问题陈述**，Plan CEO Review 审查**解决方案计划**。

建议流程：
1. 先运行 `/office-hours` 定义问题
2. 再运行 `/plan-ceo-review` 审查实现计划
