# 自动评审 Skill

## When to Use

- 每次代码改动完成后
- 任务切换时
- 用户说"评审"或"检查代码"

## 评审维度

### 1. 功能正确性

| 检查项 | 说明 |
|--------|------|
| 功能完整性 | 所有需求功能都已实现 |
| 逻辑正确性 | 业务逻辑是否正确 |
| 边界处理 | 边界情况是否处理 |
| 错误处理 | 异常是否正确处理 |

### 2. 代码质量

| 检查项 | 说明 |
|--------|------|
| 代码规范 | 符合项目编码规范 |
| 命名规范 | 变量/函数命名清晰 |
| 代码复杂度 | 函数不过于复杂 |
| 重复代码 | 无重复代码块 |
| 注释适当 | 关键逻辑有注释 |

### 3. 安全检查

| 检查项 | 说明 |
|--------|------|
| SQL注入 | 无SQL注入风险 |
| XSS | 无跨站脚本风险 |
| 认证授权 | 权限控制正确 |
| 敏感信息 | 无硬编码密钥/密码 |
| 输入验证 | 用户输入有验证 |

### 4. 性能考虑

| 检查项 | 说明 |
|--------|------|
| 数据库查询 | 无N+1查询问题 |
| 循环效率 | 无不必要的循环 |
| 缓存使用 | 适当使用缓存 |
| 资源释放 | 资源正确释放 |

### 5. 可维护性

| 检查项 | 说明 |
|--------|------|
| 模块化 | 代码组织合理 |
| 依赖管理 | 依赖清晰 |
| 可测试性 | 代码可测试 |
| 文档 | 关键代码有文档 |

## 评审流程

### 第一步：收集改动

```
改动文件：
- src/user/login.js (新增)
- src/user/validator.js (修改)
```

### 第二步：逐文件评审

**每个文件检查：**

```javascript
// login.js 评审结果

✅ 通过项：
- 功能完整
- 错误处理完善
- 命名规范

⚠️ 建议项：
- [建议] 密码验证逻辑可抽取为独立函数
- [建议] 可添加登录日志

❌ 问题项：
- [问题] 硬编码了 API Key（严重）
```

### 第三步：生成报告

```
═══════════════════════════════════════
🔍 代码评审报告
═══════════════════════════════════════

文件：src/user/login.js
─────────────────────────────────────
✅ 通过：8项
⚠️ 建议：2项
❌ 问题：1项

【❌ 严重问题】
1. 硬编码 API Key
   位置：第23行
   建议：使用环境变量

【⚠️ 建议】
1. 可抽取密码验证函数
2. 可添加登录日志

─────────────────────────────────────
评审结论：⚠️ 需修复后继续
═══════════════════════════════════════
```

### 第四步：自动修复

**对于可自动修复的问题：**
- 代码格式问题 → 自动修复
- 命名不规范 → 尝试自动修复
- 简单错误 → 自动修复

**需要人工确认的问题：**
- 安全相关问题
- 架构调整
- 业务逻辑变更

## 评审级别

| 级别 | 符号 | 含义 | 行动 |
|------|------|------|------|
| 严重 | 🔴 | 必须修复 | 暂停执行，修复后继续 |
| 重要 | 🟡 | 建议修复 | 修复后继续，或记录 |
| 轻微 | 🔵 | 可忽略 | 记录，不阻塞 |

## 触发条件

**自动触发：**
1. 每次代码改动完成后
2. 任务切换时
3. 进入审查阶段前

**手动触发：**
- 用户说"评审"
- 用户说"检查代码"
