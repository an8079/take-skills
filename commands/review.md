# /review - 代码审查

对代码进行全面的质量审查。

## 使用方式

```
/review
```

或

```
代码审查
审查代码
```

## 审查范围

| 类型 | 说明 |
|------|------|
| 代码质量 | 函数长度、文件大小、重复代码 |
| 命名规范 | 变量、函数、类的命名 |
| 错误处理 | try/catch、错误信息、边界检查 |
| 安全检查 | 硬编码密钥、SQL 注入、XSS |
| 性能考虑 | 算法复杂度、N+1 查询、缓存 |
| 最佳实践 | 设计模式、代码风格、注释 |

## 审查流程

1. 调用 **reviewer agent** 进行审查
2. 分析改动代码
3. 对照检查清单进行检查
4. 按优先级组织发现的问题
5. 生成审查报告

## 问题严重级别

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| **Critical** | 必须立即修复 | 阻止合并/部署 |
| **Important** | 应该尽快修复 | 建议修复后再合并 |
| **Suggestion** | 建议改进 | 可选修复 |

## 审查结果

### ✅ 批准

- 无 Critical 级别问题
- Important 级别问题已确认处理方案
- 代码符合项目规范

### ⚠️ 有条件批准

- Critical 问题已修复
- Important 问题有处理计划
- 建议在后续 PR 中处理 Suggestion

### ❌ 阻止

- 存在 Critical 级别问题
- Important 问题过多

## 审查通过后的下一步

```
/security      # 安全审查
/build         # 构建项目
```

---

**提示：** 代码审查不仅是找问题，更是学习机会。建设性的反馈能帮助团队共同进步。
